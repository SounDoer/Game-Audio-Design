# Game Audio Design 站点重构备忘

本文档记录重构背景、现状审计结论、目标需求与待决事项，供后续迭代对照。随讨论推进可继续增删改。

---

## 一、背景与动机

1. **原定位**：使用 Docusaurus 做「在线课程」形态的文档站，持续写入 `docs/` 下的 Markdown。
2. **动机变化**：在 AI 辅助阅读/写作普及的语境下，弱化「固定课程框架」心智，更偏向 **作者持续更新的文章/笔记集合**。
3. **内容形态期望**：回到 **以 Markdown 为源、维护成本低** 的工作流；框架是否保留 Docusaurus 可再议，但 **源文件应以 md 为主、易迁移** 为原则。

---

## 二、当前技术栈与仓库结构（审计快照）

| 项目 | 说明 |
|------|------|
| 框架 | Docusaurus 3.7，`@docusaurus/preset-classic` |
| 内容根 | `docs/`，且 `routeBasePath: '/'`（整站即文档） |
| 主要目录 | `docs/Course`、`docs/Topic`、`docs/Resource`，另有 `docs/Log.md` |
| 导航 | 三个 autogenerated 侧边栏（Course / Topic / Resource）+ 导航栏入口 |
| Frontmatter | 多为 `sidebar_position`；未见系统化的 SEO / 分享图字段 |
| 正文 | Markdown + Docusaurus admonition（如 `:::info`）+ 相对路径插图 |
| 静态资源 | `static/`（含 logo、favicon 等） |
| 本地草稿 | `_local/`（WIP、旧备份等），默认不参与 Docusaurus 构建 |
| CMS 配置 | 根目录 `frontmatter.json`（Front Matter 插件，指向 Docusaurus + `docs`） |
| Node | `>= 18`；CI 使用 Node 18 |
| 部署 | GitHub Actions：`yarn install` → `yarn build` → `peaceiris/actions-gh-pages` 发布；需仓库 Secret **`GH_PAT`** |
| 域名 | `CNAME` → `gad.soundoer.com`；`docusaurus.config.js` 中 `url` 为 `http://gad.soundoer.com`（后续可统一 **HTTPS** 与 canonical） |

### 已知问题 / 技术债（待验证）

- **`sidebars.js`**：文件中同时存在 `export default sidebars`（空对象）与 `module.exports = { courseSidebar, ... }`。需确认 Docusaurus 实际解析的是哪一份，避免侧边栏与预期不一致。
- **社交预览图**：`themeConfig.image` 仍为默认的 `img/docusaurus-social-card.jpg`，未按品牌或单篇定制。
- **分享与 SEO**：缺少每篇或全站统一的 **Open Graph / Twitter Card** 策略（标题、摘要、图）。

## 三、重构目标（需求清单）

### 3.1 信息架构与形态

- [ ] 弱化「在线课程」产品感，强化 **文章 / 系列** 或可平铺的目录与索引（具体导航方案待定）。
- [ ] 保持 **Markdown 为单一事实来源**，持续更新成本低；插图与资源路径在迁移时一次性梳理。

### 3.2 订阅与触达

- [ ] **RSS**：新文章或更新可通过订阅器触达（构建时生成 feed 或等价物）。
- [ ] **邮件订阅**：更新可通过邮件通知（需第三方 ESP 或自建后端；GitHub Pages 纯静态无法单独可靠处理订阅逻辑，通常表单对接外部服务）。

### 3.3 分享与「公众号式」预览

- [ ] 每篇文章具备 **可用的分享入口**（至少：复制链接；视平台补充二维码等）。
- [ ] **微信等 IM 内链接预览**：依赖服务端返回 HTML 中的 **Open Graph**（`og:title`、`og:description`、`og:image` 等）；需在技术方案中保证每篇有稳定 URL 与合适摘要/图。
- [ ] **预期管理**：移动端从浏览器 **直接调起「发朋友圈 / 小红书发布」** 受平台与 App 限制，多数场景为 **复制链接 + OG 卡片**，而非与原生 App 完全一致的分享 SDK 体验。

## 四、可选技术方向（未决）

以下仅作备忘，实施前需选定路径：

1. **保留 Docusaurus，瘦身**：加 RSS 插件、SEO/OG 组件、订阅表单；改动面相对可控，框架仍较重。
2. **迁移至更轻量 SSG**（如 Astro、Eleventy）：`content/` 存 md，构建 HTML + RSS；布局层统一注入 OG。
3. **极简**：仓库以 md 为主，索引/订阅由单独页面或服务承担（维护成本与「站点感」需权衡）。

**基础设施备忘**：若引入 **Cloudflare**（Pages/Workers），可在边缘处理部分订阅跳转或缓存；非必须。

## 五、待与作者确认的事项

- [ ] 托管是否仍为 **GitHub Pages + gad.soundoer.com**，是否愿意叠加 **Cloudflare** 等。
- [ ] 邮件：**纯第三方表单**（零运维） vs **自建 + 双重确认/合规**。
- [ ] 分享：**是否每篇强制独立 `og:image`**（工作量大、微信预览最稳）vs 默认图 + 标题摘要。
- [ ] **多语言**：当前配置含 `en` / `zh-Hans`；重构后是否收敛为单一语言或保留双语。
- [ ] **Docusaurus 特有语法**（admonition 等）是否必须保留，或可改为标准/通用 md 扩展。

## 六、建议实施顺序（草案）

1. 定栈与托管（是否迁出 Docusaurus）。
2. 统一 **URL、HTTPS、canonical** 与全站默认 OG。
3. 上线 **RSS**（与内容排序/「更新」定义一致）。
4. 接入 **邮件订阅**（服务商选型 + 隐私与退订文案）。
5. 单篇 **分享区 + per-page OG**（含可选封面图流程）。
6. 信息架构改版（导航、首页、Course/Topic 命名是否调整）。
